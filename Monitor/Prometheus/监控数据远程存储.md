```bash
Prometheus内置基于本地存储的时间序列数据库
在设计上使用本地存储可降低Prometheus部署和管理的复杂度同时减少高可用（HA）带来的复杂性。
默认情况下用户只需部署多套Prometheus，采集相同Targets即可实现基本HA
同时由于Promethus高效的数据处理能力，单个Prometheus Server基本上能够应对大部分用户监控规模的需求

本地存储也带来了一些不好的地方:
首先是数据持久化问题，特别是像Kubernetes这样的动态集群环境下如果Promthues实例被重新调度那所有历史监控数据都会丢失 
其次本地存储意味着Prometheus不适合保存大量历史数据 (推荐只保留几周或几个月)
最后本地存储也导致Prometheus无法进行弹性扩展
为了适应这方面需求Prometheus提供remote_write/remote_read特性，支持将数据存储到远端和从远端读取数据
通过监控与数据分离，能更好地进行弹性扩展

由于Prometheus基于Pull模型，当有大量的Target需采集时单一Prometheus实例在数据抓取时可能会出现性能问题
联邦集群的特性可以让Prometheus将样本采集任务划分到不同的Prometheus实例中
并通过一个统一的中心节点进行聚合，从而可以使Prometheuse可以根据规模进行扩展
```
```yaml
remote_write:
  - url: "http://localhost:9201/write"

remote_read:
  - url: "http://localhost:9201/read"
```
